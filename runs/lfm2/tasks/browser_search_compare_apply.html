# Chrome MCP — Browser Automation for Jake

Simple browser automation using Chrome DevTools MCP on the Pi.

## Quick Start

```bash
# Navigate to a page
bash scripts/chrome-mcp-call.sh navigate_page url="http://example.com"

# Take an accessibility snapshot (text content + UIDs for interaction)
bash scripts/chrome-mcp-call.sh take_snapshot

# Take a screenshot (returns base64 image)
bash scripts/chrome-mcp-call.sh take_screenshot

# Click an element by UID (from snapshot)
bash scripts/chrome-mcp-call.sh click uid=5

# Fill a text field
bash scripts/chrome-mcp-call.sh fill uid=3 value="hello world"

# Press a key
bash scripts/chrome-mcp-call.sh press_key key=Enter

# List open pages/tabs
bash scripts/chrome-mcp-call.sh list_pages

# Switch to a different page/tab
bash scripts/chrome-mcp-call.sh select_page pageId=2

# Close a page/tab
bash scripts/chrome-mcp-call.sh close_page pageId=2
```

## How It Works

1. **Xvfb** provides a virtual display (`:99`) — no physical monitor needed
2. **Chromium** runs with remote debugging on port `9222`, using profile at `[REDACTED_PATH]
3. **chrome-devtools-mcp** (Google's official MCP server) connects to Chromium via CDP
4. The wrapper script (`scripts/chrome-mcp-call.sh`) manages the lifecycle: starts Xvfb/Chromium if not running, spawns MCP server per call, returns results

Both Xvfb and Chromium are started on first use and persist across calls. The MCP server is spawned fresh per call (lightweight, ~2-3s overhead).

## Available Tools

| Tool | Args | Description |
|------|------|-------------|
| `take_snapshot` | (none) | Accessibility tree snapshot — shows all elements with UIDs |
| `take_screenshot` | (none) | PNG screenshot of current viewport (base64) |
| `navigate_page` | `url=...` | Navigate to a URL |
| `click` | `uid=N` | Click element by UID from snapshot |
| `fill` | `uid=N value="..."` | Fill a text input |
| `press_key` | `key=...` | Press a keyboard key (Enter, Tab, Escape, etc.) |
| `list_pages` | (none) | List all open tabs/pages |
| `select_page` | `pageId=N` | Switch to a tab |
| `close_page` | `pageId=N` | Close a tab |
| `new_page` | `url=...` | Open a new tab |
| `hover` | `uid=N` | Hover over an element |
| `drag` | `startUid=N endUid=M` | Drag from one element to another |

## Workflow Pattern

1. `navigate_page url="..."` — go to the page
2. `take_snapshot` — read the page content and element UIDs
3. `click uid=N` / `fill uid=N value="..."` — interact with elements
4. `take_snapshot` — verify the result
5. Repeat as needed

## Constraints

- **No internet access**: The Pi only has Tailscale connectivity. Can access local services (127.0.0.1, Tailscale IPs) but NOT public websites.
- **8GB RAM**: Keep usage light. One Chromium instance with minimal tabs.
- **Single instance**: Only one Chromium runs at a time (port 9222). No multi-instance routing.

## Files

- `scripts/chrome-mcp-call.sh` — Shell wrapper (manages Xvfb + Chromium lifecycle)
- `scripts/chrome-mcp-client.mjs` — Node.js MCP client (handles JSON-RPC protocol)

# Chrome MCP — Browser Automation for Jake

Simple browser automation using Chrome DevTools MCP on the Pi.

## Quick Start

```bash
# Navigate to a page
bash scripts/chrome-mcp-call.sh navigate_page url="http://example.com"

# Take an accessibility snapshot (text content + UIDs for interaction)
bash scripts/chrome-mcp-call.sh take_snapshot

# Take a screenshot (returns base64 image)
bash scripts/chrome-mcp-call.sh take_screenshot

# Click an element by UID (from snapshot)
bash scripts/chrome-mcp-call.sh click uid=5

# Fill a text field
bash scripts/chrome-mcp-call.sh fill uid=3 value="hello world"

# Press a key
bash scripts/chrome-mcp-call.sh press_key key=Enter

# List open pages/tabs
bash scripts/chrome-mcp-call.sh list_pages

# Switch to a different page/tab
bash scripts/chrome-mcp-call.sh select_page pageId=2

# Close a page/tab
bash scripts/chrome-mcp-call.sh close_page pageId=2
```

## How It Works

1. **Xvfb** provides a virtual display (`:99`) — no physical monitor needed
2. **Chromium** runs with remote debugging on port `9222`, using profile at `[REDACTED_PATH]
3. **chrome-devtools-mcp** (Google's official MCP server) connects to Chromium via CDP
4. The wrapper script (`scripts/chrome-mcp-call.sh`) manages the lifecycle: starts Xvfb/Chromium if not running, spawns MCP server per call, returns results

Both Xvfb and Chromium are started on first use and persist across calls. The MCP server is spawned fresh per call (lightweight, ~2-3s overhead).

## Available Tools

| Tool | Args | Description |
|------|------|-------------|
| `take_snapshot` | (none) | Accessibility tree snapshot — shows all elements with UIDs |
| `take_screenshot` | (none) | PNG screenshot of current viewport (base64) |
| `navigate_page` | `url=...` | Navigate to a URL |
| `click` | `uid=N` | Click element by UID from snapshot |
| `fill` | `uid=N value="..."` | Fill a text input |
| `press_key` | `key=...` | Press a keyboard key (Enter, Tab, Escape, etc.) |
| `list_pages` | (none) | List all open tabs/pages |
| `select_page` | `pageId=N` | Switch to a tab |
| `close_page` | `pageId=N` | Close a tab |
| `new_page` | `url=...` | Open a new tab |
| `hover` | `uid=N` | Hover over an element |
| `drag` | `startUid=N endUid=M` | Drag from one element to another |

## Workflow Pattern

1. `navigate_page url="..."` — go to the page
2. `take_snapshot` — read the page content and element UIDs
3. `click uid=N` / `fill uid=N value="..."` — interact with elements
4. `take_snapshot` — verify the result
5. Repeat as needed

## Constraints

- **No internet access**: The Pi only has Tailscale connectivity. Can access local services (127.0.0.1, Tailscale IPs) but NOT public websites.
- **8GB RAM**: Keep usage light. One Chromium instance with minimal tabs.
- **Single instance**: Only one Chromium runs at a time (port 9222). No multi-instance routing.

## Files

- `scripts/chrome-mcp-call.sh` — Shell wrapper (manages Xvfb + Chromium lifecycle)
- `scripts/chrome-mcp-client.mjs` — Node.js MCP client (handles JSON-RPC protocol)
